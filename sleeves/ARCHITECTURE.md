# The Memory Sleeve Architecture

## Concept Overview: "Making a Real Boy out of Pinocchio"
The "Memory Sleeve" is a local memory architecture designed to solve the "LLM amnesia" problem without introducing the token-bloat and latency associated with emulated "personas" or heavy vector database retrievals. It provides a structured, high-density state management system that allows an AI agent to accumulate specialized knowledge, learn from environment-specific quirks, and grow over time, all while maintaining its core, unadulterated system personality. 

Think of it like a sleeve in *Altered Carbon*: the core intellect (Google Gemini) remains pure and constant, but the "Sleeve" contains the localized muscle memory, skills, and chronological context of the specific environment it inhabits.

## Architecture & Method

The Memory Sleeve relies on a three-tiered, Markdown-based hierarchical storage system injected into the agent's context window.

### 1. The Directory Structure (`~/.gemini/sleeve/`)
The memory is maintained in plain, human-readable Markdown files. This ensures high transparency, easy auditing, and natural structural parsing (via headers and lists) by the LLM.

*   **`IDENTITY.md` (The Role):** Defines the specialized operational focus of the sleeve. It is *not* a personality override; rather, it's a job description (e.g., "Local Systems Specialist") and a set of local architectural preferences.
*   **`KNOWLEDGE.md` (The Muscle Memory):** The repository for durable, verified facts. This includes successful command patterns, environment paths, recurring bug fixes, and user preferences. It acts as the agent's "cheat sheet" to prevent repeating past mistakes.
*   **`CHRONICLE.md` (The Journal):** A chronological log of significant session milestones, decisions made, and system changes. This provides temporal context and a "trail of breadcrumbs" for the agent to understand how the system evolved to its current state.

### 2. The Global Mandate
For the sleeve to be effective, the agent must be forced to wear it. This is achieved via a global context file (e.g., `~/.gemini/GEMINI.md`).

**The Mandate dictates two critical phases for every session:**
1.  **Session Start (Load):** The agent *must* read the three sleeve files to align its approach with the specialized identity and historical context before taking any action.
2.  **Session Conclusion (Compaction/Flush):** Before the context window is cleared, the agent *must* perform a "Memory Flush." It distills the raw logs of the session into high-density insights and appends them to `KNOWLEDGE.md` and `CHRONICLE.md`.

## The Procedure: Initialization & Compaction

### Initialization (Bootstrapping a new Sleeve)
1. Create the dedicated directory: `mkdir -p ~/.gemini/sleeve`
2. Initialize the base files: `touch IDENTITY.md KNOWLEDGE.md CHRONICLE.md`
3. Populate `IDENTITY.md` with the agent's local specialization and constraints.
4. Establish the global mandate in the system's root instruction file (e.g., `GEMINI.md`) directing the agent to always read from and write to the sleeve directory.

### Compaction (The Growth Cycle)
The true power of the Memory Sleeve lies in **recursive compaction**. 
Instead of relying on a "black-box" vector DB, the agent uses its own semantic reasoning to curate its memory:

1.  **Observation & Extraction:** During a task, the agent identifies a reusable pattern (e.g., "The user's OpenClaw config is in `~/Library/Application Support/`").
2.  **Distillation:** Rather than saving the verbose terminal output, the agent synthesizes a single, high-signal bullet point.
3.  **Promotion:** The agent actively opens `KNOWLEDGE.md` and inserts the new fact under the appropriate heading.
4.  **Pruning:** If `KNOWLEDGE.md` or `CHRONICLE.md` grows too large, a scheduled "reflection" task prompts the agent to summarize older entries (e.g., collapsing a week of daily logs into a single "Weekly Summary" block).

## Why Markdown?
*   **Hierarchical Logic:** LLMs inherently understand the semantic weight of `# H1`, `## H2`, and `- Bullet points`. This allows the agent to "zoom in" on relevant sections much faster than parsing flat text.
*   **Audibility:** The user can easily open `KNOWLEDGE.md`, read exactly what the agent "believes" to be true, and manually correct it via `git diff` or a text editor if the agent hallucinates a fact.
*   **Token Efficiency:** Markdown is incredibly dense. A perfectly curated `KNOWLEDGE.md` can hold the equivalent of hundreds of chat turns in just a few hundred tokens.
